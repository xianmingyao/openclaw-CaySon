## 1. 🎯 核心理念

### 两个理论的共鸣

| Karpathy 工作流 | LangChain 三层 | 融合价值 |
|----------------|----------------|---------|
| raw → wiki 编译 | Harness 层优化 | 优化**执行方式** |
| wiki 查询 + 反哺 | Context 层学习 | **运行时**信息管理 |
| 知识内化（未来） | Model 层写入 | **知识写进权重** |

**共同核心：** Trace 是学习的原料，可观测性是基础设施

## 3. 🔄 完整流程

### Phase 1：收集（Collection）

```
论文/文章/GitHub/数据集/截图
        ↓
  Obsidian Web Clipper
        ↓
     raw/ 目录
        ↓
     .md + 图片
```

### Phase 2：编译（Compilation）

```
raw/ 目录
        ↓
  LLM 编译任务：
  - 提取概念
  - 写文章
  - 建分类目录
  - 加双向链接
        ↓
    wiki/ 知识库
        ↓
   Obsidian 查看
```

### Phase 3：查询（Query）

```
用户问题
        ↓
  LLM 在 wiki 中查找
        ↓
      答案
        ↓
   好的结果 ──► 反哺 wiki
        ↓
   wiki 越来越完善
```

### Phase 4：持续学习（Continual Learning）

```
Trace（执行轨迹）
        │
        ├──► Model 层：知识写进权重（未来方向）
        │
        ├──► Harness 层：优化执行框架
        │
        └──► Context 层：调整运行时配置
```

## 4. 💡 核心洞察

### 为什么不需要 RAG？

> 100 篇/40 万字规模下，LLM 上下文窗口足够大，直接把所有 wiki 内容放进 context 就够了。

```
传统 RAG：
用户问题 → 检索 → 生成 → 答案
             ↑
         (外部知识)

Karpathy 方式：
用户问题 → LLM 直接在 wiki 中查找 → 答案
             ↑
         (内化知识)
```

### 知识库的自我完善

```
查询 → 找到答案 → 好的结果归档回 wiki
                        ↓
              wiki 越来越完善
                        ↓
              后续查询越来越准
                        ↓
              LLM 定期检查数据
```

## 5. 🏢 企业应用建议

### 权限边界考量

| 层级 | 权限风险 | 建议 |
|------|---------|------|
| Model | 高（涉及模型权重） | 集中管理，谨慎更新 |
| Harness | 中（涉及执行代码） | 统一框架，分离关注 |
| Context | 低（运行时配置） | 按用户/租户分离 |

### 实施优先级

```
第一阶段：Context 层
├── 建立 Skills 机制
├── 建立 Memory 机制
└── 快速见效

第二阶段：Harness 层
├── 优化 Prompt 逻辑
├── 工具调用编排
└── 工作流自动化

第三阶段：Model 层（可选）
├── 微调特定任务
└── 突破能力上限
```

## 6. 🔧 技术实现要点

### Obsidian Web Clipper 配置

```
插件：Obsidian Web Clipper
转换：网页 → .md + 图片
存储：raw/ 目录
```

### 编译 Prompt 示例

```markdown
你是一个知识整理专家。请将 raw/ 目录下的所有资料编译成一个 wiki 知识库。

要求：
1. 提取每个文件的核心概念
2. 为每个概念写一篇简短的解释文章
3. 建立分类目录结构
4. 添加双向链接（bidirectional links）关联相关概念
5. 输出格式：Markdown

请开始编译。
```

### 查询 Prompt 示例

```markdown
你是一个知识库助手。请在 wiki/ 目录中查找以下问题的答案：
[用户问题]

如果找到答案，请提供。
如果找到相关但不完全匹配的内容，也请提供。
如果没有找到，请说明。
```

## 7. ⚠️ 风险与挑战

### 技术风险

| 风险 | 说明 | 应对 |
|------|------|------|
| 编译质量 | LLM 编译可能出错 | 人工审核 + 反馈机制 |
| 知识污染 | 错误信息积累 | 定期清理 + 版本控制 |
| 规模限制 | 超过 100 篇后性能下降 | 分层索引 + 混合方案 |

### 工程风险

| 风险 | 说明 | 应对 |
|------|------|------|
| 维护成本 | wiki 需要持续更新 | 自动化 + 人工结合 |
| 上下文耗尽 | LLM 上下文有限 | 智能压缩 + 检索增强 |
