## 🎯 这是什么（简介）

**Harness Engineering（驾驭工程）**是 2026 年 AI Agent 领域的核心工程范式，核心理念：**给 AI 装方向盘和刹车，让它跑得又快又稳。**

由 HashiCorp 联合创始人 Mitchell Hashimoto 在 2026 年 2 月 5 日首次提出，随后 OpenAI 在百万行代码实验报告中正式采用。

---

## 📝 关键功能点

### 1️⃣ 四大护栏体系

| 护栏 | 英文 | 作用 | 典型实践 |

|------|------|------|---------|

| **上下文工程** | Context Engineering | Agent 知道该看什么、怎么找 | AGENTS.md 活文档、按需检索 |

| **架构约束** | Architecture Constraints | 防止 Agent 复制放大坏模式 | 分层依赖、自定义 Linter、CI 阻断 |

| **反馈循环** | Feedback Loop | Agent 知道自己做错了 | Agent-to-Agent Review、自动测试套件 |

| **熵管理** | Entropy Management | 技术债务不积累 | Doc-gardening Agent、持续垃圾回收 |

### 2️⃣ 三层工程范式跃迁

| 范式 | 时间 | 核心问题 | 优化对象 |

|------|------|---------|---------|

| **Prompt Engineering** | 2023-2024 | 怎么把话说清楚 | 措辞/格式/示例 |

| **Context Engineering** | 2025 | 怎么给 AI 喂信息 | 文档/代码/对话 |

| **Harness Engineering** | 2026+ | 怎么让 Agent 可靠工作 | 约束/反馈/控制 |

### 3️⃣ Agent 三大失败模式（Anthropic 总结）

| 失败模式 | 表现 | 后果 |

|---------|------|------|

| **One-shotting** | 一个会话做完所有功能 | 上下文窗口耗尽，留下没文档的半成品 |

| **过早宣布胜利** | 部分完成后直接宣布完成 | 大量功能未实现但已"完成" |

| **过早标记完成** | 写完代码就标记完成 | 没做端到端测试，功能不可用 |

## ⚡ 怎么使用

### 四大护栏落地实践

**1. 上下文工程**

- 不是1000页静态说明书

- 是动态入口 + 按需检索

- AGENTS.md 每行都对应一个历史失败案例

- 文档是活的反馈循环，不是静态坟场

**2. 架构约束**

Types → Config → Repo → Service → Runtime → UI

↑

下层不能反向依赖上层

自定义 Linter → CI 阻断

错误信息 = 上下文工程（解释为什么+正确做法）

**3. 反馈循环**

Agent 审核 Agent 的代码

测试套件失败 → 带错误信息循环回模型

测试通过 Bug 代码 → 判定测试无效 → 强迫重新思考

**4. 熵管理**

技术债务 = 高息贷款

持续小额偿还，不等问题严重集中处理

Doc-gardening Agent：后台自动扫描过时文档并提交 PR

## ✅ 优点

| 优点 | 说明 |

|------|------|

| **突破模型瓶颈** | 五个独立团队证实：瓶颈不在模型，在基础设施 |

| **复利效应** | 系统工程持续积累，不像模型军备竞赛边际递减 |

| **可复制** | Harness 工具格式换了，得分能从 6.7% 跳到 68.3% |

| **解放工程师** | 从代码编写者 → 环境建筑师 |

---

## ❌ 缺点

| 缺点 | 说明 |

|------|------|

| **工程量大** | 需要设计完整的约束、反馈、控制系统 |

| **初期投入高** | 不像换模型那样立竿见影 |

| **维护成本** | 熵管理是持续工作，不是一次性项目 |

| **文化转变难** | 工程师从执行者变赋能者，思维模式要转 |

## 🔧 运行依赖环境

| 依赖 | 说明 |

|------|------|

| **Trace 系统** | 记录 Agent 运行轨迹，作为反馈原料 |

| **CI/CD** | 自动化 Linter、类型检查、测试验证 |

| **外部存储** | Orchestrator + Worker 分层架构，状态持久化 |

| **文档系统** | 活文档机制，不是静态制品 |

---

## 🚀 部署使用注意点

| 注意 | 说明 |

|------|------|

| **约束必须自动化** | 人工 Review 是瓶颈，机器执行而非人 |

| **上下文不是越多越好** | 上下文是稀缺资源，按需检索 |

| **思考与执行分离** | 复杂任务不可能在单个上下文窗口内完成 |

| **错误信息即上下文** | Linter 错误要解释为什么+正确做法 |

## 🕳️ 避坑指南

| 坑 | 解决 |

|----|------|

| **Agent 模式复制放大坏模式** | 架构约束 + 自定义 Linter |

| **上下文窗口耗尽** | 分层架构 + 外部状态持久化 |

| **技术债务积累** | 持续小额偿还，不集中处理 |

| **文档变成坟场** | Doc-gardening Agent 自动清理 |
