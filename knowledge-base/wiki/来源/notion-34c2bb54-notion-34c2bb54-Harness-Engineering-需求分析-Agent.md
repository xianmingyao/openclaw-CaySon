## 1. 核心概念

### 1.1 上下文工程（Context Engineering）

**定义：** 在有限的上下文窗口中，选择、组织并注入与用户输入或任务高度相关的信息，让大模型能够在合理的边界内做出最佳推理和执行。

**核心原则：** 用最相关的信息填充 LLM 的上下文窗口

### 1.2 Harness Engineering

**定义：** 为 Agent 搭建运行空间，设计它的能力结构、协作机制和反馈闭环，让它在特定领域里稳定地产生高质量结果。

**核心区别于"束缚"：**

- ❌ 束缚 = 控制每一步执行，让 Agent 只能沿骨架行走

- ✅ 创造 = 定义边界和协作协议，创造条件让模型能做到原本做不到的事

**关键洞察：**

> Harness Engineering 真正在做的事情是**定义边界和协作协议**，而不是控制每一步的执行。

> 它不是在限制模型能做什么，而是在**创造条件让模型能做到原本做不到的事**。

## 2. MrMaKer 的需求分析 Agent 三板斧

基于 Harness 思想构建的需求分析 Agent，核心就是**三点**：

### 2.1 架构约束（Architectural Constraints）

**做什么：** 把要求写清楚，规则写得细致

**目的：** 明确边界，让 Agent 知道能做什么、不能做什么

**实操：**

- 需求文档必须包含：背景、目标、范围、功能点、非功能需求、验收标准

- 规则细化到：输出格式、术语定义、约束条件

- 不给 Agent 留模糊空间

**示例：**

❌ 模糊："做一个好看的界面"

✅ 约束："登录页使用 #2563EB 品牌色，标题 24px 正文字体 16px，圆角 8px"

### 2.2 环境闭环（Environment Loop）

**做什么：** 每个阶段都设计审核环节，没通过就重新生成，直到通过才进入下一阶段

**目的：** 通过反馈循环确保质量，不让错误传递到下一阶段

**流程：**

[生成] → [审核] → [通过?] ─┬─ 是 → [下一阶段]

│

└─ 否 → [根据审核结果重新生成] → [审核] ...

**审核要点：**

- 完整性：是否覆盖所有需求点？

- 一致性：是否与上一阶段产物冲突？

- 可执行性：技术上能实现吗？

### 2.3 知识治理（Knowledge Governance）

**做什么：** 每阶段的产物只需要根据上一阶段的产物生成

**目的：** 避免知识混乱，确保追溯链清晰

**原则：**

- 阶段1需求文档 ← 用户原始需求

- 阶段2功能设计 ← 阶段1需求文档（不再看原始需求）

- 阶段3技术方案 ← 阶段2功能设计（不再看需求文档）

- ...

**好处：**

- 避免 Agent "遗忘" 上下文

- 清晰的传承链

- 易于定位问题在哪一阶段

## 3. 需求分析 Agent 的五阶段模型

结合 Harness Engineering，可以设计这样的五阶段：

| 阶段 | 输入 | 产出 | 审核要点 |

|------|------|------|----------|

| **1. 需求收集** | 用户原始需求 | 需求文档 | 完整性、清晰度 |

| **2. 需求分析** | 需求文档 | 功能列表 | 一致性、优先级 |

| **3. 流程设计** | 功能列表 | 流程图/时序图 | 逻辑完整性 |

| **4. 方案设计** | 流程设计 | 技术方案 | 可行性、性能 |

| **5. 验收标准** | 方案设计 | 验收测试用例 | 可测试性 |
